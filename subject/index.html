어떤 데이터를 활용해야 하는가?

어떤 서비스를 제공하려 하는가?

지하철과 버스, 택시에서 발견된 분실물들을 검색할 수 있게 해주는 서비스

api적용과 실제 사용이 매우 난잡하여 같은 역할을 하는 다른 api로 교체하였습니다.




사용자와 과업 분석

1~4호선은 코레일과 서울메트로 소속으로 나뉘는 구간이 있으며, 5~8호선은 서울도시철도공사소속이다.
하지만 5~8호선은 네이버에 노출되는 서울메트로 홈페이지 에서 바로 찾을 수 있는 반면에, 1~4호선은 분실물이 발생한다면 네이버 검색에선 찾기가 힘들다.
또한, 버스나 택시에서 잃어버린 분실물들은 못찾는다고 생각하는 사람이 대다수 인점을 생각하면 이 서비스는 나름의 해결책이 될듯 하다.
그래서 버스나 택시에서 잃어버린 분실물 정보와 홈페이지 한 곳에서 1~4호선과 5~8호선 분실물을 찾을 수 있다면 매우 편리하게 이용할 수 있을것 같다.

    • 1. 가설 설정

  • 사용자가 빠른 시간 내에 자신이 잃어 버린 분실물의 정보를 찾을 수 있는가.

    • 2. 과업 설정

  • 지하철(1~4호선) 분실물 정보 얻어오기
  • 택시 분실물 정보 얻어오기

    • 총 2명에게 사용자 평가를 진행하였다.
    주로 자가용 차를 소유하고 있지 않은 대학생을 평가 대상에 포함시켰다.

    "서울시 열린 데이터 광장"의 api 질이 국내에선 최고급이라 파싱하는데엔 별다른 어려움이 없었다. 그래서 데이터가 매우 빠르고 깔끔하게 나오게 되었다.
    또한 html의 기본 table 태그는 디자인이 맘에 들지 않아, w3school에서 css 코드를 참조하여 나름 깔끔하게 꾸며보았다.
    하지만 흠이라면, 1~4호선 데이터를 불러올 때 앞에 ":"문자열이 추가되어서 파싱되기 때문에 조금 보기 좋진 않다. 하지만, 최종버전때 split함수로 처리하여 깔끔하게 출력할 예정이다.
    하지만 코드가 상당히 더러워질 것 같은 느낌이다.
    실험에 앞서, 유튜브에 업로드 하는 것에 대한 동의를 받았으며 어느 용도로 사용될 것인지에 대한 동의를 받았다.

    첫 번째 참가자는 지하철 2호선에서 아이폰6를 잃어버렸다고 가정하여 실험을 진행하였다.
    데이터의 질이 좋아서 빨리 찾을 수 있었다. 또한, 아주 직관적이라는 평을 들었다. (단순히 분실물 목록만 보여주는 형태였기 때문에 직관적이지 않으면 안될것 같았다.)
    한 가지 아쉬운 점이 있다면, 이 물품이 내가 잃어버린 물품이 맞는지 확인할 수 없어서 사진도 같이 볼 수 있다면 좋겠다는 평도 있었다.

    두 번째 참가자는 개인택시에서 가야금을 잃어버렸다고 가정하여 실험을 진행하였다.
    이 참가자에게서 역시 '직관적으로 보기 쉽게 되어있는가'에 대해선 "너무나 직관적이다." 라는 평가를 받았으며,
    '빨리 찾을 수 있는가'에 대해선 ctrl+f 키를 눌러서 찾아야 했기에 조금은 불편했다는 평가를 받았다.
    또한, '가야금'이 특수한 물품이라 잃어버리면 목록에서 바로 찾기는 쉽겠지만, 스마트폰이나 배낭 같이 일반적으로 잘 잃어버리는 물품에 대해선 첫 번째 참가자와 마찬가지로 사진이 포함되었으면 좋겠다는 평을 듣기도 했다.
    그리고, 모든 참가자에게서 디자인이 너무 이쁘다는 평을 받기도 했었다. 직관적인가 - O O
    찾기에 편리한가 - ▲ ▲
    다음 최종본엔 검색 기능과 분실물 이미지 출력 기능을 추가해야 더욱더 편리하게 쓸 수 있을것 같다.






    사용자 대상 실험_3 최종 완성본

    이번엔 최종 결과물인 만큼, 여러 api를 적용해보려고 하였다.
    특히, 검색기능의 api와 이미지 api를 넣을려고 하였지만, 검색 기능을 넣으려 한다면, 기존 코드를 거의 완전히 갈아엎으면서 몇 가지 기능을 포기해야 하는 경우가 생기기에, 적용하지 않았다.
    그리고 분실물 이미지 api는 서울시 열린 데이터 광장에 사용하기 쉽게 게재되어 있지만,
    실제로 버스, 택시, 지하철 모두가 분실물 담당 업무를 하는 사람이 사진을 찍어서 올리지 않아, api를 적용해도 사진이 출력되지 않는 경우가 90%이기 때문에, 허울뿐인 api는 제외하는것이 더 낫다는 판단이 들어 제외하기로 했다.

    그래서 이 많은 분실물들의 목록을 어떻게 친근하게 다가갈가를 고민하던 차에, 가장 익숙한 불편함을 떠올렸다.
    기존 대화형 프로토타입에선 단순히 테이블 목록만 띄워주고 끝을 냈었다.
    하지만, 목록이 너무 많기 때문에 가끔씩 피곤하면 약간의 난독증이 찾아오는 나같은 사람에겐 줄을 따라 읽다가 다음 줄로 넘어가는 경우가 생기곤 했다. 그리하여, 가장 간단하게 css문법으로 구현할 수 있지만,
    그나마 데이터를 좀 더 쉽게 받아들일 수 있는 테이블의 "hover"기능을 탑재하여 가독성을 높였다.
    몇몇 사람들에게 이 프로토타입의 전 후를 비교하며 보여주고, 사용성이 확 달라진 느낌을 받았다는 평가도 받기도 했다.

    먼저 실험영상을 보도록 하자.


    총 5명에게 실험을 진행하였다. 대체로 깔끔하고 간결하다는 평가를 받았다.

    실험은 특정 물건을 잃어버린 가상의 상황에서, 진행되었다. 그리고 저번 실험 결과와 다른 점은, 대화형 프로토타입때의 실험을 마치고 나서, 눈을 돌린 참가자들이 다시 자신이 찾던 물품을 찾아볼 시에, 못찾고 다시 처음부터 스크롤을 하며 읽어 내려가는 현상이 있었지만
    최종 결과물 실험 영상을 찍고 나서 참가자들이 자신이 찾던 물품 테이블위에 hover가 적용된 것을 보고 다시 쉽게 찾는 현상도 발견하였다.
    마치 운영체제의 버퍼 캐쉬를 구현한 듯한 느낌이 들기도 했다.
    아쉬운 점으로는 우리나라의 공무원들이 제대로 일을 하지 않아 api적용을 하지 못한점이 아쉽다고 평가하기도 했다.
    또한, 이번에는 처음으로 외부 학교의 인문계 학생에게 테스트를 진행해보았다.
    그 결과 생각지도 못한 버그가 발생하였다.
    그 동안 줄곧, 우리학교의 컴소과를 대상으로만 진행해서 모두가 크롬에 익숙해있었지만, 다른 과는 그렇지 않았다.
    대부분 IE를 쓰기때문에 내 웹페이지가 작동을 잘 하지 못하였다. IE를 전혀고려하지 않은 명백한 내 잘못이었다.
    IE용과 크롬용 웹페이지를 따로 만들기도 한다는데 진짜였던 것이다.

    요번 과제를 진행하면서 두고두고 아쉬울 점은 분실물 이미지 API를 적용하지 못한것이 가장 아쉬움이 크다.
    간신히 테이블에 표시된 분실물의 이름마다 분실물 ID에 맞게 이미지 Hyperlink를 달아주는 작업을 하였으나, 공무원들의 근무태만으로 실패한 것이 아주 마음에 걸린다.
    지금까지는 과제가 잘 되지 않는다면, 1%는 visual studio와 불안정한 우분투 환경, 99%는 내 잘못이었지만,

    내 잘못이 아닌 다른 사람에 의해 과제 진행이 되지 않는건 신박한 경험이었으며, 한국의 API관리 실태를 몸으로 느낀 사례가 될 것 같다.
    향후 개선 방향으로는 다른 웹 페이지 템플릿을 적용하여, 이렇게 보고서가 포함된 결과물이 아닌, 순수한 분실물 찾기 서비스를 탑재한 웹 페이지를 호스팅하는 것이다.
    또한, 분실물 목록 테이블을 한 페이지에 쭉 표시하는 방법이 아닌, 일정 개수 만큼 잘라서 1, 2,3, 4, 5, ... N 페이지 만큼 생성하여 서비스 해야할 것 같다.
